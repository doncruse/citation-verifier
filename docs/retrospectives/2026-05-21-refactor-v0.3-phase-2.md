## Surprises (what the plan didn't survive contact with code)

### S1. The plan's `_process_citation_lookup_hit` example was internally inconsistent

The plan showed `_process_citation_lookup_hit(builder, t, ...)` calling `self._finalize_result(builder, ...)` *inside* the `with builder.stage(...)` block. But `_finalize_result` reads `builder.entries()`, and `builder.stage()`'s context manager only appends the entry in its `finally` clause. So the plan's example would have returned a `VerificationResult` with the current stage's entry missing from the path.

**Resolution:** The Task 2 implementer changed `_process_citation_lookup_hit`'s return type from `VerificationResult` to a kwargs dict (`cluster_id`, `absolute_url`, `text_source`, `warnings`, `status`). Callers exit the `with` block first, then call `_finalize_result(builder, **kwargs)`. This is the only ordering that produces correct path entries. The constraint is documented in the helper's docstring; call-site comments at `verify()`, `verify_async()`, `verify_batch()` would have helped readers but were flagged as nice-to-have, not blocker.

**For Phase 3 plan-writing:** when an example uses a builder/accumulator pattern with a deferred append, exercise the example mentally (or trace the entries list at each line) before committing it to the plan. The "fixture before assertion" rule applies to plan code too.

### S2. Task 3 introduced an orphaned-resolved-entry bug that Task 4 had to fix

When `_search_fallback` runs opinion_search first (and it resolves at e.g. 0.83), then `_build_fallback_result` sees the parsed citation has no court AND no year, it flips the final status to NOT_FOUND. But the opinion_search resolved entry stays in the path. `headline_confidence` (which reads the last resolved/partial entry) returned 0.83 even though `status == NOT_FOUND`. The existing async parity test (`test_parity_insufficient_data_guard`) caught this because Task 4 also activated the async per-stage path; sync tests didn't catch it because Phase 1's `_build_fallback_result` was monolithic.

**Resolution:** The Task 4 implementer hoisted the "no court and no year" guard to the top of `_search_fallback` AND `_search_fallback_async`, short-circuiting before any network call. The redundant guard in `_build_fallback_result` is left in place as defense-in-depth, which is a minor CLAUDE.md anti-pattern violation ("don't add error handling for scenarios that can't happen") — but the cost of removing it is the next refactor possibly forgetting why the guard moved. Acceptable.

The implementer's scope-expansion judgment was good: catching a regression that only surfaced once the async side also reached parity is exactly the integration test loop working as intended.

**For Phase 3 plan-writing:** when per-stage instrumentation could create "early-stage success, late-stage NOT_FOUND" patterns, the plan should explicitly call out which guards need to short-circuit before stage runs (vs. which guards can be evaluated post-pool). The Phase 2 plan didn't flag this and Task 3 broke it; Task 4's parity test caught it.

### S3. Task 5 was almost entirely already done by Task 2

The plan budgeted Task 5 as a meaningful instrumentation task: "verify_batch creates a per-citation `ResolutionPathBuilder` for each batch hit and miss." But Task 2's implementation of `verify_batch` already created per-citation builders (the alternative — calling the deleted `_build_result` — would have crashed). Task 5 collapsed into a tests-only commit (82 lines added, all tests).

This is not a bug in Phase 2 — it's a fortuitous side effect of doing the structural refactor right in Task 2. But the plan's per-task budgeting overshot.

**For Phase 3 plan-writing:** when a headline-flagged task says "instrument X" but X was already touched in an earlier task's structural work, drop the headline flag and rebudget. The "size each task by the production-code delta it ships" heuristic worked better than "size each task by the bullet point count."

### S4. The `"via": "batch"` extra key was a small spec drift caught and rolled forward

The Task 2 spec reviewer flagged that `verify_batch`'s `raw_response_summary` carried an extra `"via": "batch"` key not in the plan's pinned shapes table. The Task 5 implementer reviewed whether to remove it; the resolution was "keep it" because Task 6's `REQUIRED_KEYS` test uses subset semantics (`required - entry.raw_response_summary.keys()`), so extras are tolerated, and `"via"` is useful debug context that distinguishes batch-path entries from single-citation entries.

**For Phase 3 plan-writing:** when a plan pins a `raw_response_summary` shape, explicitly state whether the test is exact-equality or subset. The Phase 2 plan accidentally got this right (subset) but the spec reviewer had to do the analysis to determine that. A one-line "Schema is minimum-required, extras tolerated" note in the table preamble would save the question.

### S5. The `notes` flow decision was left to the implementer and they made the right call

The plan's Task 3 explicitly left the notes-flow design open (lines 1190–1196: "Don't duplicate the notes onto a synthetic terminal entry — that would re-introduce the Phase 1 anti-pattern" but offered two implementation choices). The Task 3 implementer chose to update the test-side `_diagnostics` / `_matched_case_name` helpers to find the winning entry by `_winning_entry(path)` (highest-confidence resolved/partial else last), rather than mutate the builder's entries post-pooling.

This was the right call:
1. No production-code mutation of "frozen" path entries.
2. The classifier helper is destined for deletion in Phase 3 anyway (per Phase 1 retrospective Q2 — WarningCategory promotion), so investing in the production-side mutation API would have been wasted.
3. `_winning_entry` is a small, well-defined helper that captures the "which stage actually resolved this citation" question that consumers will eventually ask in production code too.

**Note for Phase 3:** when WarningCategory promotion lands and `_diagnostics` / `_classify_note` get deleted, `_winning_entry` may remain useful as a public helper. Consider moving it from `tests/test_verifier.py` into a `tests/_compat.py` module shared between test files (Phase 1 retrospective note 4 already flagged the consolidation).

### S6. Cache round-trip worked without changes

Phase 1's `_serialize_path_entry` / `_hydrate_path_entry` in `cache.py` already handled all `ResolutionPathEntry` fields correctly. Task 8's 2 tests passed without any production change.

**For Phase 3:** when Phase 3 adds the `caption_investigation` stage or extends `ResolutionPathEntry` fields, run `tests/test_cache_roundtrip.py` early. If the existing helpers are field-aware (introspect via dataclass fields), Phase 3 may also need no cache.py changes — but Phase 3 should verify, not assume.

## Open questions to fold into Phase 3's plan

### Q1. `_winning_entry`'s home and signature

The Phase 2 implementation of `_winning_entry(path)` lives in `tests/test_verifier.py`. The production code doesn't have a parallel helper because nothing in the current `VerificationResult` API needs one (`headline_confidence` already provides the value-of-interest). Phase 3 should decide whether:
- The helper stays test-only (fine — its role is purely to support the legacy `_diagnostics`/`_classify_note` compat helpers).
- It becomes a `VerificationResult` method (e.g. `result.winning_entry` returning `ResolutionPathEntry | None`).
- It's deleted entirely once `_diagnostics`/`_classify_note` go away (most likely).

### Q2. The 0.40 threshold magic constant

After Task 3, the `0.40` "VERIFIED" score threshold is duplicated across at least 5 sites: `_search_fallback` (each of the three stage blocks), `_stage_notes_for_candidate`, and `_build_fallback_result`. The Task 3 code reviewer flagged this for Phase 3 extraction. Suggested form: a module-level `_VERIFIED_SCORE_THRESHOLD = 0.40` constant, possibly tunable.

When Phase 3 audits diagnostic policy (per the §2.6 WarningCategory promotion work), it's a natural moment to also extract this constant. Avoid extracting it as a refactor-in-passing during caption_investigation work — that's two unrelated changes in one commit.

### Q3. The `partial` verdict is unused in Phase 2

`StageVerdict.partial` is defined and supported by the builder but no production code path emits it. Phase 3's `VERIFIED_PARTIAL` status will need at least one entry-emission path to use it. Likely candidates:
- citation_lookup name-mismatch (currently emits `resolved` at 0.30 confidence with a `cl_display_name_data_bug` warning) — could move to `partial`.
- The opinion_search "name-overlap-but-not-good-enough" middle-ground (currently a no_match below 0.40) — could become a `partial` band.

Phase 3 should explicitly define which (stage, score) bands emit `partial` so the §2.5 partial-verdict contract is honored consistently across the three stages.

### Q4. `caption_investigation` stage entry-point

The plan's design v2 §2.5 reserves `StageName.caption_investigation` for a Phase 3 stage that distinguishes CL display-name data bugs (stays VERIFIED) from genuine WRONG_CASE. Per Phase 2's instrumentation pattern, this stage would be opened *inside* `_process_citation_lookup_hit` when a name mismatch is detected — between the `t.resolved(...)` call and the `_finalize_result` call. Phase 3's plan should specify:
- Whether caption_investigation is a sibling stage to citation_lookup (opens its own `builder.stage(...)` block) or a sub-step within citation_lookup (mutates the citation_lookup entry's `raw_response_summary` or notes).
- What `raw_response_summary` shape it emits.
- Whether it can ALSO be triggered after opinion_search resolutions (e.g., when opinion_search returns a high-scoring candidate but the name fundamentally disagrees).

### Q5. The `cl_display_name_data_bug` warning lives, but is not yet refined

Task 2 ships a citation_lookup name-mismatch path that emits a `Warning(category=cl_display_name_data_bug, ...)` and stays VERIFIED at 0.30 confidence. Phase 3's caption_investigation stage may want to:
- Promote some name-mismatch cases to WRONG_CASE (when the caption truly diverges, not just stylistically).
- Keep others as VERIFIED + cl_display_name_data_bug (when the caption is consistent under caption rules but CL's display name is stale).

Phase 3 needs to define the gates that drive this decision. Without it, every name-mismatch citation_lookup hit stays VERIFIED at 0.30 forever, which is *better* than Phase 1's behavior (which silently dropped the mismatch into LIKELY_REAL) but not yet correct.

## Notes for whoever writes the Phase 3 plan

1. **Read this retrospective AND Phase 1's retrospective AND the design v2 doc.** Phase 3 builds on top of Phase 2's instrumentation, which builds on Phase 1's schema. None of the three documents alone is sufficient context.

2. **Plan the caption_investigation stage explicitly.** Don't leave its entry point ambiguous — name the function it lives in, name the cluster IDs/names it'd query, decide whether it opens a sibling `builder.stage(...)` block or mutates an existing one (Q4 above).

3. **Bundle the WarningCategory promotion + `_diagnostics` classifier deletion + `_winning_entry` consolidation into one task.** They're all the same conceptual cleanup — moving from "free-form notes parsed by regex" to "closed-set categories on Warning objects."

4. **0.40 threshold extraction is a 10-line cleanup.** Schedule it inside another task, not as its own.

5. **Don't add new path-emission paths during caption_investigation work.** Phase 2's instrumentation is now load-bearing for every consumer; adding a new stage with a new `raw_response_summary` shape needs an explicit row in the per-stage shapes table (and a `REQUIRED_KEYS` entry).

6. **The `live_api` deselect pattern works.** Keep it. The CLI smoke for Obergefell is fast enough (~2s) to be the single live-API check during day-to-day work; the full live-API suite (currently 1 test, ~35s) only needs to run at acceptance gates.

7. **Phase 3 plan should be written AFTER reading this file.** If the implementer reads only the design v2 doc, they'll miss S2 (Task-3 regression caught only by Task 4), S5 (notes-flow design choice) and Q4 (caption_investigation's exact entry point).
