## Surprises (what the design didn't survive contact with code)

### S1. `cache.py` was missing from the plan's "Modified" list

The plan listed `cache.py` under "Not touched in Phase 1," but `cache.py` imports `VerificationStatus` and constructs old-shape `VerificationResult` for cache hydration. Once `models.py` rewrote those, `cache.py` failed to import — and `cache.py` is imported by both `__main__.py` and `web/app.py`. So the moment Task 1 landed, *everything* downstream broke harder than the plan anticipated.

Resolution: I folded cache.py into Task 2 as a scope-correction. The implementer replaced `asdict()` / `dict(**)` round-trip with explicit `_to_dict`/`_from_dict` helpers that walk nested dataclasses and re-hydrate Enums. Public API unchanged. This was the right call — cache.py is small enough to migrate cleanly inside Task 2, and any alternative ("disable cache temporarily") would have created its own breakage.

**For Phase 2 plan-writing:** before listing files as "not touched," `grep -l 'VerificationStatus\|VerificationResult'` across `src/` to catch this kind of dependency.

### S2. `__init__.py` had to drop the `CitationVerifier` re-export mid-migration

Task 1 landed with `models.py` rewritten but `verifier.py` still on the old shape. That meant `from citation_verifier import CitationVerifier` failed at import time. To get `tests/test_models.py` collectable, I had to comment out the `CitationVerifier` re-export in `__init__.py` and restore it in Task 2 after verifier.py was healthy.

**For Phase 2 plan-writing:** if a phase plans to rewrite a downstream module across multiple tasks, expect the package's `__init__.py` re-export of that module's public name to need temporary suspension. List this explicitly in the plan.

### S3. Task 7's scope was understated

The plan said Task 7 was just the confidence-print line in `test_false_negatives.py`. The file actually had ~10 old-shape touchpoints (import of `VerificationStatus`, status check, `result.matched_cluster_id`, `result.matched_case_name`, `result.matched_url`, `result.diagnostics`, `result.confidence`). I did them all in one inline edit. Not a big deal — but the plan's "tiny task" was actually a small task.

**For Phase 2 plan-writing:** when listing a per-file task as "tiny," grep the file first to confirm.

### S4. `test_cl_api_issues.py` had one stray `result.matched_case_name`

The plan listed `test_cl_api_issues.py` as "Audit-by-running, not by-rewriting" — and the file mostly is. But one test (`test_our_abbreviation_workaround`) reads `result.matched_case_name` directly. Caught during the Task 9 acceptance gate; one-edit fix. Could have been caught during Task 4 if I'd run *every* test file at that point instead of just the ones in the plan's scope.

**For Phase 2 plan-writing:** at each task boundary, run the full test suite and triage anything red — not just the suite of test files in the task's scope.

### S5. Working-directory `.env` not auto-populated in the worktree

The verify-brief acceptance test (Task 9 Step 4) initially failed with 401 Unauthorized on every citation-lookup call. Root cause: `client.py` does `load_dotenv(_env_path)` where `_env_path = Path(__file__).parent.parent.parent / ".env"`. In a git worktree, `parent.parent.parent` is the worktree root, which (because `.env` is gitignored) doesn't have an `.env` file — the real one lives in the primary checkout. Single citation queries worked via anonymous access; batch queries required auth and failed.

Fix: copied `.env` from the parent checkout into the worktree root. Not a refactor regression; a worktree-setup gap.

**For the project CLAUDE.md "Refactor Workflow" section:** add a note that each worktree needs its own `.env` (or symlink) for live-API tests to pass.

### S6. The Obergefell test in `test_false_negatives.py` is flaky on rate-limit

During the Task 9 full-suite run, the Obergefell live-API test failed once with a transient CL hiccup; passed cleanly when re-run alone. Not a refactor regression — this is the existing live-API surface's normal flakiness. The 1-second rate-limit + the test_async_verifier batch tests + smoke calls during this session occasionally collide with the live API's throttling.

**For Phase 2 acceptance:** consider deselecting `live_api`-marked tests during the full-suite acceptance run, and run them separately at a slower cadence.

## Open questions to fold into Phase 2's plan

### Q1. Is `_build_result` good enough, or does Phase 2 need a different shape?

Phase 1's `_build_result(stage, verdict, ...)` produces a single `ResolutionPathEntry`. Phase 2 wraps *every* attempted stage and produces multiple entries. The helper's "one entry" API is the wrong shape for Phase 2 — it needs a builder that accumulates entries across a verification run. **Suggested Phase 2 shape:** a `ResolutionPathBuilder` context-manager-ish object that the verifier instantiates once per `verify()` call and appends to as each stage runs. The `_build_result` helper would become the *terminal* call that wraps the accumulated path into a `VerificationResult`.

### Q2. The diagnostic→warning bridge in `verifier.py` (joined into `notes`)

The code-review fix-up's prefix-anchored classifier in test_verifier.py works, but the production-side bridge — `notes = "; ".join(d.message for d in diagnostics)` — loses the per-diagnostic category. Phase 3's design (per design §2.6 amendment workflow) considers adding closed-set `WarningCategory` values for the legacy categories (court mismatch, date mismatch, docket mismatch, RECAP, etc.). When that lands, the test-side classifier helper can be deleted.

**For Phase 3's plan:** explicitly enumerate which Diagnostic categories from the legacy verifier should promote to WarningCategory enum members. Re-audit the prefix-anchored regex table in `tests/test_verifier.py:42-70` and `tests/test_async_verifier.py` at the same time.

### Q3. The cluster-ID drift xfails in `test_false_negatives.py`

The 4 xfailed cases (Anderson, Bossart, Busha, Townsley) were CL-data drift, not verifier regressions. Phase 1 didn't touch them. None of them flipped to green or red during the migration. **Phase 3 caption-investigation work** is the natural place to revisit these — if Phase 3 introduces full-caption investigation (per design §2.5 caption_investigation stage), the cluster-ID drift cases may resolve to the new IDs via that path. Pop them off xfail when that happens.

### Q4. The `confidence=0.95` vs `confidence=1.00` discrepancy in single-citation runs

`python -m citation_verifier "Obergefell..."` returned `Confidence: 95%`. Wave1 batch returned `Confidence: 1.00`. Why the diff?

The single-citation path goes through the sync `verify()` -> citation-lookup -> if no match, falls to opinion-search-fallback at 0.95. The batch path goes through `_batch_citation_lookup` directly. For Obergefell, the batch found a citation-lookup hit (1.0) but the single path didn't?

This is a pre-existing oddity, not a Phase 1 regression. **For Phase 2:** instrument the resolution_path entries deeply enough to see *which* stage resolved a given citation, and assert sync/async/batch parity. The Phase 2 instrumentation already plans for this.

### Q5. The CSV schema's `syllabus` column

`verification_results.csv` now writes an empty `syllabus` column. The plan's disposition was "Phase 3 re-evaluates whether to add it to FinalIds." Two related questions for Phase 3:

- Should `FinalIds` carry the matched cluster's syllabus, or should syllabus stay on a `raw_response_summary` for the citation_lookup stage (more in keeping with the §2.5 free-form-per-stage rule)?
- The legacy `result.matched_syllabus` was used by `metadata_check` for "syllabus consistency" — that path now reads `claims.csv` via the merge step. Phase 3 should confirm the metadata_check path is correct (no current tests broke during Phase 1, but the coverage may be thinner than ideal).

## Notes for whoever writes the Phase 2 plan

1. **Pre-plan the audit grep.** Before writing the file-modified list, run `grep -l 'VerificationStatus\|VerificationResult\(' src/` AND grep through any private (`_foo`) names that the schema-rewrite touches. Anything that constructs the old shape needs to migrate.

2. **Plan for the `_build_result` rewrite.** Phase 2 instruments every stage. The helper's signature changes. Schedule that change up-front in the plan, not as a refactor-in-passing during a per-stage instrumentation task.

3. **Live-API tests are gated on `.env`.** Document the `.env` requirement in setup. Consider adding a `conftest.py` skip marker that auto-skips live-API tests if no token is set.

4. **The `_diagnostics()` test-side bridge in test_verifier.py and test_async_verifier.py** is duplicate code. Phase 2 (or whenever the WarningCategory promotion happens) should consolidate into `tests/_compat.py` and have both test files import it. Don't ship that during Phase 2 if it adds scope; just file it.

5. **Phase 2's plan should be written *after* this retrospective is read.** If the implementer reads only the design v2 doc and not this retrospective, they'll miss S1 (cache.py), S2 (__init__.py), S5 (.env), and the Q1 helper-shape problem.
