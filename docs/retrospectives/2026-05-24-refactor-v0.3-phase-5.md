## Surprises

### A1's actual broken line was not what the audit named

The audit said the regression was at `web/app.py:1475` (`_process_citation_lookup_hit` v0.2 signature). When Task 1's harness ran, the actual failure traceback pointed at `web/app.py:1515` (`_search_fallback_async` v0.2 signature). Both are real, but the route happened to hit the latter first when the stub returned empty batch results — every citation went into miss_indices and immediately TypeErrored on the fallback call.

**Lesson:** The audit was right about the class of regression but wrong about the specific call site. The harness caught it without depending on the audit's line accuracy, which is the right shape for this kind of test — *any* ERROR-status event under the stub is a signature-shape regression.

### The plan's `test_post_qc_run_batch_no_typeerror` assertion was buggy

The plan's test body wrapped the assertion in `if event == "result" and data.get("status") == "ERROR": assert "TypeError" not in err`. Python's `str(TypeError(...))` doesn't include the literal word "TypeError" — it's just the message — so the assertion never fired even when the route was raising TypeErrors. Caught and fixed during Task 1 self-review: the right assertion is `data.get("status") != "ERROR"` because the all-empty stub never produces real errors; any ERROR is a signature regression.

### `verify_batch()` has its own latent bug

Task 2's first attempt routed `/api/qc/run-batch` through `verify_batch()`. That surfaced a separate bug: `verifier.py:2890` splats `hit_finalize` into `_finalize_result` without popping the `needs_caption_investigation` / `_cluster_for_investigation` keys that `_process_citation_lookup_hit` adds. The `verify()` and `verify_async()` paths handle them at lines 697 and 2377; `verify_batch` was missed when Phase 3's caption_investigation work landed.

**Disposition:** Task 2 took the plan's conservative-fallback path (per-citation `verify_async` for the QC endpoint), same posture as `/api/verify`'s addendum fix. The `verify_batch` bug is now tracked in `docs/consumer-surface-manifest.md` "Known issues" for Phase 6+.

### `_STATUS_DISPLAY` is dead code

Audit confirmed only 1 reference in `web/app.py` (its own definition). Implementer's call per plan: left in place; documented in manifest for Phase 6+ cleanup.

### Exit-code precedence: INCOMPLETE wins, not NOT_FOUND

The plan's code had `max()` semantics with NOT_FOUND=1, INCOMPLETE=2 — which means INCOMPLETE wins when both appear in a batch. The plan's comment claimed the opposite ("NOT_FOUND beats INCOMPLETE (1 < 2 is intentional)"). Caught during Task 8 test-writing. The code's behavior is actually defensible ("if anything was incomplete, you can't fully trust the NOT_FOUND signal — retry first"); fixed the comment to match the code, added `test_incomplete_beats_not_found` to pin the behavior.

### Frontend test regex needed brace-balancing

The plan's coarse regex for extracting JS switch bodies tripped on nested block-scoped `case 'POSSIBLE_MATCH': { ... }` blocks in `statusBadges()` — it stopped at the first `}}`. Rewrote as a brace-counter that tracks string/comment context. Slightly more code than the plan's regex but actually robust.

### Function naming is per-page

Plan assumed all three HTML files had `statusLabel` + `badgeClass`. Actual: get.html has only `statusBadges`; index.html has all three; qc.html has `statusLabel` + `badgeClass`. Parametrization updated accordingly.
