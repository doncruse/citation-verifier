## Surprises (what the plan didn't survive contact with code or data)

### S1. The procedural-keyword list rejected substantive opinions

Task 4's `_PROCEDURAL_KEYWORDS` initially included `"motion for"` and `"motion to"` to catch obviously-procedural docs like motions in limine. The implementer caught and removed both entries — they appear as substrings inside legitimately-substantive descriptions like "ORDER GRANTING Motion for Reconsideration" or "OPINION on motion to dismiss." The fix was a code comment ("intentionally excluded") plus removal from the keyword tuple.

But Task 6's live-API run still found a related issue: the opinion-typed keyword list doesn't include `"order granting"` / `"order denying"`. Mehar Holdings' substantive 12-page opinion has description "ORDER GRANTING 14 Motion for Reconsideration re 13 Order" — matches no opinion keyword, falls to DOCKET_ONLY under strict gating. Documented as Phase 4 follow-up: either expand keyword lists or add page-count heuristics.

**For Phase 4 plan-writing:** keyword-only opinion-typing is fragile. Real descriptions vary widely — "ORDER GRANTING X", "MEMORANDUM AND ORDER on Y", "Findings of Fact and Conclusions of Law", "Decision". A page-count + free-on-pacer fallback would catch the substantive-but-keyword-poor cases.

### S2. The lenient `_names_match_citation_lookup` masks "X v. United States" caption divergence

Named exemplar `named-exemplar-koch` was the canonical cl_display_name_data_bug test: brief cites "Koch v. United States, 857 F.3d 267 (5th Cir. 2017)"; CL returns cluster 4390987 named "Ricky Koch v. Tote, Incorporated". The design expected: VERIFIED + `cl_display_name_data_bug` warning fires after caption_investigation.

But caption_investigation never fired. `_names_match_citation_lookup` accepts "Koch" as a lenient surname-match (the function uses surname containment, and "Koch" IS in "Ricky Koch v. Tote, Incorporated"). The "X v. United States" pattern with a distinctive plaintiff passes the lenient check even when the defendant differs completely from CL's caption.

The `_names_match_citation_lookup` common-prefix branch handles "United States v. X" (US as plaintiff) but not "X v. United States" (US as defendant). The asymmetry is the bug.

**For Phase 4:** extend the common-prefix detection to recognize generic-government-defendant patterns (`v. United States`, `v. State`, `v. Commonwealth`, etc.) and require defendant-side overlap in those cases. Six Phase 3 fixtures (Koch + 5 Rule-25d/SSA-pseudonym fixtures) lost their expected `cl_display_name_data_bug` warning because of this gap, and the survey §3.1 documents each one's `phase3_ruling`.

### S3. `cl_display_name_data_bug` only fires on citation_lookup hits, not on opinion_search resolutions

A related finding: the rule-25d and SSA-pseudonym fixtures resolve via opinion_search (not citation_lookup), because CL's citation index doesn't carry the WL cite in many of these cases. The divergence between brief caption and CL caption IS detected — visible in the opinion_search stage notes ("Name mismatch: cited 'X' vs found 'Y'") — but no typed `WarningCategory` is emitted, because caption_investigation only runs after citation_lookup.

Affected: `verified-rule-25d-gilliard-mcwilliams`, `verified-rule-25d-preston-smith`, `verified-rule-25d-viken-detection`, `verified-ssa-pseudonym-john-s-bisignano`, `verified-ssa-pseudonym-michael-b-berryhill`.

**For Phase 4:** extend caption_investigation (or add a parallel post-opinion_search hook) to emit `cl_display_name_data_bug` when opinion_search resolves with a divergent CL case_name. Equivalent gate: party-overlap on (brief.case_name, opinion_search-resolved cluster's case_name) — if overlap holds, emit the warning; if not, escalate to WRONG_CASE.

### S4. The strict ±14-day window penalizes WL-only citations without specific dates

`verified-via-recap-doe-lawrence`'s citation is `Doe v. Lawrence Gen. Hosp., 2025 WL 2808055` — no specific date in the citation text. `parse_citation()` extracts `year=2025` but `month=None`, `day=None`. The verifier's date-proximity check defaults cited date to June 15 (mid-year), but the RECAP doc is filed Aug 29 — outside the ±14 day window. Result: DOCKET_ONLY despite the doc being labeled "Report and Recommendations" (opinion-typed).

**For Phase 4:** either (a) widen the date window when citation has no specific date, or (b) accept VIA_RECAP if opinion-typing is unambiguous, regardless of date proximity, or (c) some hybrid (e.g., wider window only when description matches strong opinion keywords). The strict ±14 days only makes sense when the citation actually carries a specific date to compare against.

### S5. RECAP fuzzy-rescue is too lenient on confirmed-fake citations

`not-found-gibbs-wright` is a court-confirmed hallucinated citation (from the Hardy v. Ford sanctions order). Phase 3 still returns VERIFIED_DOCKET_ONLY because `recap_docket_search` finds a docket on a fuzzy name+date match. The case is fake, but RECAP has SOME docket that approximately matches and the verifier rescues it.

This is a known weakness. The fixture was updated to expected `VERIFIED_DOCKET_ONLY` to match current behavior, with a Phase 4 follow-up flag in the survey ruling.

**For Phase 4:** consider tightening `recap_docket_search` — require an opinion-typed doc on the docket to escape NOT_FOUND, OR require higher fuzzy-match score for the docket caption. The current behavior over-rescues; the corpus's `hallucinated_case` category exposes this clearly.

### S6. The §0.3 Darensburg substitution surfaced a deeper WL-disambiguation limit

The §0.3 Westlaw lookup of `2009 WL 2392094` showed the WL number actually maps to an Aug 4 procedural costs-taxation order (doc #460), not the substantive July 7 attorneys' fees opinion (doc #452). The two docs are on the SAME docket; only Westlaw "knows" which one earned the WL number. The verifier can't disambiguate from docket data alone.

This made the Mehar Holdings substitution unstable too — Phase 3 strict gating produced DOCKET_ONLY for Mehar because its description doesn't match opinion keywords (see S1). So the VIA_RECAP named-exemplar slot is effectively empty after Phase 3.

**For Phase 4:** the WL-disambiguation limit is genuinely hard. Either the verifier needs an external WL-index lookup (not currently available in CL), or it needs to accept "we can't fully verify this WL → docket → specific-doc chain" as a known indeterminate. The current strict VIA_RECAP gate at least avoids false-positives by being conservative.

### S7. The corpus reorientation around live data made some "drift" rulings cleaner than expected

The original plan budgeted 2–3 days for Task 6 because resolving 10 phase3_classification_open fixtures sounded fiddly. In practice, the live-API dump + the rulings script + the second corpus walk took one ~3-hour session including the two 12-minute live-API passes. The dump-then-edit-then-validate loop was effective because: (a) the dump JSON exposed every fixture's actual verifier output in one document, making rulings mechanical rather than judgment-heavy; (b) the rulings script let me apply all 22 fixture updates atomically; (c) the second live-API pass either confirmed everything or surfaced one or two remaining issues to triage.

**For Phase 4:** when corpus rulings come up again, default to "dump first, edit second, validate third." The naive flow of "read fixture → run verifier → write ruling → repeat" wastes API budget and developer attention.

## Open questions to fold into Phase 4's plan

### Q1. `VERIFICATION_INCOMPLETE` production wiring (the main Phase 4 task)

Phase 3 left `VERIFICATION_INCOMPLETE` as a type that exists but is not produced. The corpus has 5 fixtures with `mock_spec` declarations for HTTP 500 / 429-exhausted / timeout / connection / malformed-JSON; Phase 4 needs to (a) build the mock-spec harness, (b) wire the verifier's internal API-error path to set `Status.VERIFICATION_INCOMPLETE`, and (c) decide whether `caption_investigation`'s defensive fallback (currently VERIFIED + `cl_display_name_data_bug` on infra failure) upgrades to `VERIFICATION_INCOMPLETE` once the gate lands.

The Phase 2.5 mock_spec schema is well-defined (per the corpus's `mock_spec.stages` field); Phase 4 implements the consumer.

### Q2. The Mehar Holdings "ORDER GRANTING" opinion-typing gap

Phase 3's strict gate rejects "ORDER GRANTING Motion for X" descriptions even when they ARE the substantive opinion. Phase 4 should either:
- Expand `_OPINION_KEYWORDS` to include `"order granting"` + `"order denying"` (risk: catches procedural "ORDER GRANTING Stipulated Protective Order")
- Add page-count heuristics (`page_count >= 5 OR is_free_on_pacer`) as a secondary gate
- Add an `opinion_likelihood` score (the existing `_pick_best_recap_doc` heuristic already produces this) and gate VIA_RECAP at a score threshold rather than keyword presence

Pick whichever the implementer finds cleanest. The retrospective vote is for the score-based gate.

### Q3. The lenient `_names_match_citation_lookup` "X v. United States" gap

Per S2 above, extend the common-prefix detection so caption_investigation fires for Koch and similar fixtures. Six fixtures need this for `cl_display_name_data_bug` to start firing; Phase 4 should update their `phase3_ruling` notes and re-add `cl_display_name_data_bug` to `expected_warnings_subset` once the fix is in.

### Q4. The `wrong_page_number` warning category is dead code under current corpus

Phase 3 added `WarningCategory.wrong_page_number` per maintainer Q2 but no fixture exercises it. Butler Motors was the candidate, but neither cited page resolved, so the warning has nothing to fire against. Phase 4 should either:
- Find or synthesize a fixture where one page resolves and a sibling cluster at a different page exists.
- Or document the category as forward-looking infrastructure for a use case that hasn't surfaced yet.

If Phase 4 doesn't surface a fixture, consider whether the category should be deprecated. The CHANGELOG entry says it's added; the §2.6 amendment workflow says removals are a major version bump, so this is a deliberate decision.

### Q5. The `cl_duplicate_clusters` warning fires only when both lookups land at the same case_name + date

Phase 3's caption_investigation has hooks for `cl_duplicate_clusters` but the corpus exercises it only weakly. Phase 4 should confirm against the benchmark whether the 4 xfailed entries (Bossart, Busha, Townsley, Anderson-Furst) ever needed this category, or whether their drift was always a single-cluster pin-update case. The current rulings updated pins; none required `cl_duplicate_clusters`.

### Q6. The candidates field on VerificationResult

The roadmap (`scratch/ROADMAP.md`) item to surface multiple candidates when the verifier is uncertain is still open. Phase 3 expresses uncertainty only via warnings (notably `cl_duplicate_clusters` per Q5). Phase 4 should decide whether to grow the API surface with `candidates: list[CandidateMatch] | None` or keep expressing uncertainty through warnings only.
