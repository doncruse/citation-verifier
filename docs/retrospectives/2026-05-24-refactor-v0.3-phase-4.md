## Surprises (what the plan didn't survive contact with code or data)

### S1. The `MockSpecPatcher` `attempt_idx` semantics broke the rate-limit fixture

The plan's harness incremented a per-stage call counter and fired only when `count == target_attempt_idx`. But the harness wraps `_request_with_retry` (the OUTER retry loop), and the verifier calls `client.citation_lookup()` exactly once per `verify()`. The `verification-incomplete-rate-limit-exhausted` fixture has `attempt_idx: 2` — the harness would have needed 3 separate `citation_lookup` invocations before firing, which never happens.

The harness's own docstring acknowledged the intended semantics ("focuses on what the verifier sees post-exhaustion") but the code didn't honor it. Fix: drop the count gate; fire on first matching call. `attempt_idx` is preserved on the spec for documentation but no longer affects firing. This is Critical issue C1 from the Opus review of `6b15835`.

**For future:** if the design says "the retry loop is not exercised inside the harness," the harness must not condition on attempt counts.

### S2. `search_recap` returns `page_count=None` and `is_free_on_pacer=None`

The plan's Task 4 assumed both fields would be available on the docket dicts returned from `search_recap`. They aren't — those fields are only populated on the `/recap-documents/{id}/` detail endpoint. The mock tests passed because the mocks filled the fields; the live API exposed the gap when Mehar stayed DOCKET_ONLY despite the gate being correctly implemented.

The fix (option 1 in the user prompt) was to add a doc-detail fetch: when the keyword/date path fails and `is_free_on_pacer` isn't already True, fetch `/recap-documents/{id}/` to get the real metadata, then re-apply the gate. This costs one extra API call per RECAP-only no-keyword-match citation but is necessary for the gate to do its job.

A subtle refinement: the trigger condition is `not best.is_free_on_pacer` (catches both None and False), not `page_count == 0` (because `page_count` can be populated even when `is_free_on_pacer` is omitted from the search response). The implementer caught this — `not best.is_free_on_pacer` correctly fires whenever the search omits the field and is safe when `is_free` is genuinely False (the detail fetch confirms it, gate stays DOCKET_ONLY).

**For future:** when designing a gate that depends on RECAP doc metadata, audit which fields are populated by the search endpoint vs. the detail endpoint. The two surfaces are not equivalent.

### S3. Townsley fixture upgraded VIA_RECAP via the new gate, exposing a caption-divergence gap

After Task 4's score gate landed, `verified-townsley-xfailed` flipped from DOCKET_ONLY to VIA_RECAP. The cited "Townsley v. Lifewise Assurance Co." resolves to docket 5352576 — but CL's caption for that docket is "Townsley v. SNC-Lavalin Constructors, Inc.", a different defendant. The score gate accepts the doc as substantive (it IS a substantive opinion) but the verifier doesn't surface the caption divergence because `caption_investigation` only fires after `citation_lookup` resolves — not after `recap_docket_search`.

This is the same limit as Phase 3 retro S3 (opinion_search divergence detection) generalized to recap_docket_search. The fix is the same: extend caption_investigation (or add a parallel post-resolution hook) to emit `cl_display_name_data_bug` / `WRONG_CASE` based on party-overlap when opinion_search or recap_docket_search resolves with a divergent CL case_name. Out of Phase 4 scope; added to ROADMAP as Phase 5+ work alongside the opinion_search variant.

**Disposition:** Townsley fixture updated to expect VIA_RECAP with a `phase4_ruling` explaining the caption divergence and the Phase 5+ follow-up.

## Addendum: Phase 4 Task 9 pre-merge smoke surfaced 3 consumer-side regressions

After the Phase 4 retro above was written and committed (`a1adfe8`), the user asked for a deep-check of pipeline consumers before the final merge to main. Three v0.3-schema regressions surfaced in 90 minutes of ad-hoc smoke testing. Each was fixed in a separate addendum commit on `refactor/v0.3`, but the pattern they reveal — consumer-side compatibility was not systematically audited during Phases 1–3 or Phase 4 — argues that **Task 10 (merge to main) should be blocked on a Phase 5 consumer-compatibility sweep** rather than merged with what we've happened to catch.

### A1. `web/app.py` `/api/verify` batch fast path (commit `e6bf6ca`)

`CitationVerifier._process_citation_lookup_hit`'s signature changed during Phases 1–3 from `(citation_text, parsed, batch_hit)` to `(builder, token, citation_text, parsed, cluster, clusters_returned)`, and its return type changed from `VerificationResult` to a dict of `_finalize_result` kwargs. `web/app.py:422` never got updated. Every batch citation_lookup hit (the common case) raised `TypeError` and surfaced to the frontend as `status='ERROR'`. Not caught by the existing test suite because the web app has no integration tests.

Fix: drop the manual batch fast path; route every citation through `CitationVerifier.verify_async()` (the public API). Trade-off: loses the batched citation_lookup optimization (1 API call for N citations). For typical web usage (1–10 citations) negligible; for bulk workloads it would matter and Phase 5+ should restore a correct batch path using the new helper signatures.

### A2. `tests/verify_from_csv.py` v0.3 schema references (commit `6a21e5f`)

The master iterative verification workflow (per CLAUDE.md) referenced four v0.2 fields that don't exist on v0.3 `VerificationResult`:
- `result.confidence` → `result.headline_confidence` (property)
- `result.matched_url` → `result.final_ids.absolute_url`
- `result.matched_case_name` (lost; v0.3 doesn't carry it on the top level) → extract from `result.resolution_path[-1].raw_response_summary` via helper
- `result.diagnostics` → `result.warnings` (different shape: Warning vs Diagnostic)

`--dry-run` worked (skips verification), so the bug was silent under casual testing. The script would crash on the first real verification call.

### A3. Frontend JS status switches (commit `3316fd2`)

`statusBadges()` / `badgeClass()` / `statusLabel()` in `web/static/get.html`, `index.html`, and `qc.html` all had v0.2-only switch statements. All five new v0.3 statuses (`VERIFIED_PARTIAL`, `VERIFIED_VIA_RECAP`, `VERIFIED_DOCKET_ONLY`, `WRONG_CASE`, `VERIFICATION_INCOMPLETE`) hit the default branch and rendered as permanent "Searching..." / "Pending" badges. **User-visible repro: paste a citation that resolves to DOCKET_ONLY (e.g. an MDL where the specific opinion can't be pinned) → API returned `VERIFIED_DOCKET_ONLY`; UI showed "Searching..." forever.**

The user caught this manually in the browser, not via test. The same gap exists in the QC page's filter chips (still v0.2-only; not yet fixed).

### Why these slipped past Phase 4

**The Phase 4 plan named Task 8 "brief_pipeline + report_template 4-status awareness."** That was the consumer-teaching task. But its scope was narrowed to one consumer (the proposition checker), not all of them. The plan's "Not touched in Phase 4" section explicitly excluded `web/app.py`:

> `web/app.py` — Phase 4 does not touch the FastAPI server. The web app produces the new statuses correctly; presentation polish for them is roadmap.

Two errors in that one sentence:
- **"produces the new statuses correctly"** — false; the batch fast path was broken
- **"presentation polish is roadmap"** — true for genuine polish, but the missing status cases in the JS switch caused the UI to render "Searching..." forever, which is load-bearing, not polish

Task 8 should have been scoped as "every consumer of the new statuses" — brief_pipeline + report_template + web app `/api/verify` serialization + frontend JS badges + QC page filter chips + iterative CSV workflow. Instead it was just the first one, and "the web app produces them correctly" was assumed.

### Lessons for Phase 5

1. **Audit, don't smoke.** The 3 addendum fixes happened via ad-hoc smoke. A Phase 5 plan needs to enumerate every consumer surface and verify each, not pick a few and hope.
2. **"Produces correctly" needs a test, not an assumption.** The plan's confidence that the web app "produces the new statuses correctly" was wrong because nothing tested it. Add integration tests for web app endpoints during Phase 5.
3. **The frontend is a consumer too.** The Phase 4 plan implicitly treated `src/citation_verifier/` as "the codebase" and `web/app.py` as out-of-scope; `web/static/*.html` JS wasn't even mentioned. Phase 5 needs to list these explicitly.
4. **Plan scopes should match the change they're integrating.** A status-taxonomy change touches every consumer that displays or filters status. Task 8's "brief_pipeline + report_template" scope was narrower than the change demanded. Phase 5's consumer-sweep plan should be sized to the schema change, not to a single consumer.
