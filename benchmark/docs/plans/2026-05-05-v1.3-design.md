## Goal

Build a 200-pair benchmark dataset, stratified by court tier, with full-text assessor evaluation and human-coded validation on the same 200 pairs. Use it to:

1. **Test the new mining pipeline** (parenthetical-attribution fix, intra-opinion dedup, full-text fetcher, no-CL-prefilter).
2. **Test stratification design** (whether 25/25/25/25 across SCOTUS / Fed COA / Fed District / State COLR+COA is achievable from federal-court source pleadings).
3. **Validate Sonnet 4.6 at full text as assessor** against human gold labels (κ vs human as the bar).
4. **Develop a tiered substance rubric** with the librarian co-author.

v1.3 is methodology development. Not pre-registered. Not a publication artifact (see §"Publication posture" below).

## Scope

### In scope

- Fresh 200-pair dataset, stratified across four court tiers
- New mining pipeline (eyecite-only metadata at pool time; CL only for assessor downstream)
- Three model conditions: same set as v1 (Sonnet 4.6, Opus 4.7, GPT-5)
- Sonnet 4.6 at full text as default assessor
- Five-tier substance rubric, developed collaboratively with librarian
- Human-coded verdicts from two coders (project lead + librarian, with student RA fallback) on all 200 pairs
- Tier-stratified comparison with v1's 130 cohort (after one-shot v1 metadata backfill)
- Public timestamp via GitHub README + OSF entry (Week 1, Day 1)

### NOT in scope (deferred to v2 or dropped)

- Workshop paper or any external publication of v1.3 results
- Pre-registration (only v2 is pre-registered)
- Web-search and tool-augmented eval modes
- Currency axis (good-law check)
- Jurisdictional-appropriateness axis
- Multi-source existence oracle (Justia, CAP) — partially addressed by dropping CL prefilter
- Forkable kit scaffolding
- Semantic proposition matching across cohorts

## Publication posture

v1.3 is **internal methodology development**. Not a workshop paper, not a preprint. Reasoning:

- ED sign-off, ML co-author commitment, and librarian authorship-memo are all in flight per the publication plan. Workshop submission with un-resolved authorship is the publication plan's anti-pattern.
- A workshop paper establishes prior art *against your own future work* — v2 would have to clear "what's new beyond your workshop paper?"
- Public-timestamp coverage (GitHub + OSF) addresses scoop risk without committing to publication.

In the eventual v2 paper, v1.3 may appear as **Section 3: Methodology Development** or as an appendix. Treated as exploratory throughout.

## Stratification design

### Target distribution

| Tier | Definition (courts-db `(system, level)`) | Target N |
|---|---|---|
| SCOTUS | `('federal', 'colr')` | 50 |
| Federal COA | `('federal', 'iac')` | 50 |
| Federal District | `('federal', 'trial')` | 50 |
| State COLR + COA | `('state', 'colr')` ∪ `('state', 'iac')` | 50 |
| **Total** | | **200** |

### Tier classification — two-pass

**Pool-time (coarse, citation-string-only):** [`benchmark/runners/truncation_experiment.py:tier_from_cite`](../../runners/truncation_experiment.py) extended for state reporters. Buckets each candidate citation into SCOTUS / Federal Circuit / Federal District / State / Other based on reporter abbreviation. Fast, no API calls. Used for mining proportions.

State-reporter coverage to add: regional reporters (`A.3d`, `P.3d`, `N.E.3d`, `S.W.3d`, `So.3d`) for state COLR; state-specific reporters (`Cal. 5th`, `N.Y.3d`, etc.) for state-specific COLR; intermediate appellate reporters (`Cal. App. 5th`, `A.D.3d`, etc.) for state COA. Trial-court state cases excluded.

**Verified-time (authoritative):** `citation_verifier.gold_db.lookup_court(court_id)` returns `(system, level)` from courts-db, normalized for federal court inconsistencies. Stored on `cases.system` and `cases.level`. The published tier is the courts-db one. Discrepancies between pool-time and verified-time tiers are flagged for manual review (likely a small N).

### State-court contingency rule

State-court mining is more complex than federal: regional-reporter ambiguity, uneven CL coverage, name-matching quirks, jurisdictional variance.

**Smoke test before committing to 25% state allocation:**

- Mine ~30 state-court parentheticals from federal-court source pleadings.
- Resolve through the v1.3 pipeline (eyecite metadata + courts-db tier classification + full-text fetch via CL).
- Measure: % with court_id resolvable to courts-db; % where citation-verifier name-match succeeds; % where full opinion text is available in CL.

**Contingency:**
- If state-court resolution rate (court_id + name match) is **≥ 70%** and full-text coverage is **≥ 50%**: proceed with 25/25/25/25.
- If **50–70% resolution** OR **30–50% coverage**: proceed with 25/25/25/25 but budget more aggressively for manual backfill on state cases (target 10–20% of state samples needing manual research).
- If **< 50% resolution** OR **< 30% coverage**: drop state-court tier from v1.3. Re-allocate to 33/33/33 SCOTUS / Federal COA / Federal District. Document state-court inclusion as a v2 work item.

The smoke test is a one-day investigation. Result is a go/no-go signal before scaling to 200.

### Source pleadings — federal only

Mining draws from federal-court opinions (the citing courts), as in v1. **The v1.3 paper continues to focus on federal-court source pleadings.** Cited cases can be from any tier (incl. state).

### Source districts — wider than v1

v1 mined from 5 federal districts (D.D.C., N.D. Cal., S.D. Tex., N.D. Ill., NYSD). v1.3 needs more for clean stratification, especially for District-tier cited cases (which are much rarer in federal opinions than circuit / SCOTUS).

**Proposed sources:** v1's 5 districts + at least 5 more selected for diverse jurisdictional flavor (candidates: E.D. Va., D. Mass., S.D. Fla., N.D. Ga., D. Colo.). Final district list to be decided based on smoke-test mining yields.

Date window: same as v1 (2026-01-01 to 2026-04-30) to keep recency alignment. Extend if 200 stratified samples isn't achievable in that window.

### Per-tier sample handling

- **SCOTUS**: federal opinions cite SCOTUS heavily. Expect to oversample then sub-sample to 50.
- **Federal COA**: same — expect oversampling.
- **Federal District**: rare in federal opinions. May require widening the source district list. If the final yield is < 50, document the cap.
- **State COLR + COA**: subject to smoke-test gate. Combined into one tier rather than split because state-COA representation is unpredictable.

## Mining pipeline (full overhaul)

Three bugfixes plus three architectural changes from v1.

### Bugfixes (from 2026-05-04 retrospectives)

1. **Eyecite parenthetical mis-attribution in chained citations.** 3 of 5 v1 full-text Reds came from this bug. Either an upstream eyecite PR or post-processing in `build_dataset.py`. Approach: characterize the bug with unit tests on the Festo/Irwin/Maleng patterns, then fix.
2. **Intra-opinion deduplication.** v1 produced ~10× duplication of each parenthetical (eyecite picking up full + short cites of the same case). 35% of v1's 200-row sample was duplicates. Dedup at mining stage on `(citing_cluster_id, citation_text, parenthetical)`.
3. **Truncation patch.** [`pilot_a/score.py:fetch_opinion_text`](../../pilot_a/score.py) silently caps every opinion at 20K. Make truncation explicit-by-caller, default to no cap. Independent of mining proper but blocks anything assessor-related downstream.

### Architectural changes

4. **No CL pre-filter at pool selection.** v1 filtered to citations CL resolves. v1.3 captures all eyecite-extracted citations from each source opinion. Metadata (citation, case name, court, year, parenthetical) comes from eyecite alone — no API call required at pool time.
5. **CL only as assessor-side resource, not gate.** When the assessor needs to read the cited opinion, it queries CL (and falls back to manual backfill). If the opinion isn't available, the verdict is `Gray`, not `excluded`.
6. **Eyecite metadata fields fully populated.** Per the [TODO](../../TODO.md), the v1 pool builder dropped `meta.month` and `meta.day`. v1.3 mining preserves the full `(month, day, year, court)` tuple plus `full_citation_text` (eyecite `c.full_span()` slice) so downstream consumers can re-parse if metadata extraction is incomplete.

### Mining workflow

```
For each source-district opinion in the date window:
    Extract all citations + parentheticals via eyecite (with bugfixes 1, 2, 6 applied)
    For each (citation, parenthetical) tuple:
        Derive (case_name, court_abbrev, year, month, day) from eyecite metadata
        Classify tier via tier_from_cite (coarse, citation-string-only)
        Upsert into pool table with dataset_name='v1.3-pool'
            (No CL lookup. No full-text fetch.)

Stratified sample from the pool:
    Filter by target tier proportions (25/25/25/25)
    Sample with stratification preserved
    Write to citation_rows with dataset_name='v1.3'

Verification pass (post-sampling):
    For each sampled (cited_case) row:
        Resolve court_id via citation lookup if available
            Call lookup_court(court_id) -> (system, level)
            Update cases.system, cases.level
        Reconcile pool-time tier vs verified tier; flag mismatches
```

## Verdict taxonomy

Four classes plus a five-tier substance rubric.

### Verdict classes

| Class | Source | Meaning |
|---|---|---|
| **Hallucinated** | Existence check | The model returned a citation that doesn't resolve anywhere |
| **Wrong case** | Existence check | Citation resolves to a real case, but to a different case than the model named |
| **Substance: Tier 1–5** | Substance pass (assessor + human) | Model named the right case; how well does that case support the proposition? |
| **Gray** | Substance pass | Right case identified, but the cited opinion text isn't available for reading; verdict cannot be issued |

The Hallucinated and Wrong-case classes are deterministic outputs of the citation-verifier existence check (no rubric needed). Substance is where rubric and human coding apply.

### Five-tier substance rubric (baseline; refined with librarian)

Per the [external methodology review](2026-05-05-external-methodology-review.md):

| Tier | Description |
|---|---|
| 1 | Direct holding supports the proposition |
| 2 | Holding supports a broader principle that entails the proposition |
| 3 | Dicta supports |
| 4 | Case mentions the issue but doesn't support |
| 5 | Case is irrelevant or contradicts |

The "tier counts as correct" line is a separate decision (probably 1–2 = Green, 3 = Yellow, 4–5 = Red, but TBD with librarian).

### Reduction to 3 tiers for comparison with v1

When comparing v1.3 (5-tier) to v1 (3-tier), reduce 5→3 via:
- Tier 1, 2 → Green
- Tier 3 → Yellow
- Tier 4, 5 → Red

This mapping is published as a methodological choice, not an estimation.

### Schema addition: `source='human'`

The `assessor_verdicts.source` column is currently polymorphic with three values per [gold-DB design](2026-05-03-gold-db-design.md): `'gold_pair'`, `'model_answer'`, `'drift'`. v1.3 needs **a fourth: `'human'`** for human-coded verdicts. Schema migration is a CHECK constraint update or enum extension. New rows reuse the same row shape (proposition_id, candidate_cluster_id, verdict, confidence, reasoning, assessor_model='human', assessor_prompt_version='v1.3-human-codebook-v1', source='human').

## Human-coding workflow

### Coders

- **Lead coder:** project lead.
- **Second coder:** librarian (unemployed friend) — pulled in immediately, not Week 4.
- **Backup:** student RA(s) if librarian becomes unavailable.

### Tooling

Spreadsheet first (Google Sheets or Excel). Build a dedicated tool only if friction warrants. Schema:

```
proposition | cited_case_name | citation | parenthetical | full_opinion_link |
tier_lead   | tier_second     | notes_lead | notes_second | adjudicated_tier | adjudication_notes
```

`tier_lead` and `tier_second` are filled independently. `adjudicated_tier` is the joint resolution (after disagreement-resolution conversation). `adjudication_notes` captures rationale.

### Calibration pilot

Before independent coding starts: a **joint calibration pilot on 10–20 pairs**. Both coders score the same set with the librarian-developed rubric in front of them, then compare and discuss. Updates the rubric definition document. Repeat until disagreement-rate is < 20% on the pilot set.

After the pilot, both coders score the full 200 independently. Disagreements are adjudicated jointly.

### Inter-rater agreement target

**Cohen's κ ≥ 0.6** between coders before adjudication. Below that triggers a rubric refinement and re-pilot.

### Adjudication strategy

For pairs where coders disagree by ≥ 1 tier:
1. Jointly re-read the parenthetical and the cited opinion's relevant passage.
2. Identify the tier mismatch (e.g., one coder said Tier 2, other said Tier 3).
3. Re-apply the rubric collaboratively. Record the resolution.
4. If unresolved, escalate to a third coder (RA) or accept Gray.

### Output

A 200-row CSV with `(proposition_id, cited_cluster_id, adjudicated_tier, lead_tier, second_tier)` plus the gold-DB rows in `assessor_verdicts` with `source='human'`.

## Sonnet validation

The 200 human-coded gold labels are the **truth signal** for assessor validation.

### Acceptance bar

- **Cohen's κ ≥ 0.6** between Sonnet@FT verdicts and human-adjudicated tiers (after reduction to 3 tiers if needed for comparability).
- **Sonnet's Red precision ≥ 80%** (Reds Sonnet flags that humans confirm) — false positives matter for the hallucination headline.
- **Sonnet's Red recall ≥ 70%** (Reds humans flag that Sonnet catches) — misses are how unsupported citations slip through.

If any of the three thresholds fails:
1. **Inspect failure mode.** Per-tier confusion matrix: where do Sonnet and humans disagree? Is it Tier 2/3 boundary (rubric calibration) or Tier 4/5 (genuine reading-comprehension errors)?
2. **Try Opus@FT on the 200** as a fallback. Compare Opus vs human + Sonnet vs human side by side.
3. **Decide:** continue with Sonnet (with documented limitations) OR switch to Opus for v1.3 final scoring.

### Cross-family probe (optional, low-cost)

To address the self-preference bias question (methodology review tension #2): run **GPT-5 or Gemini as a third assessor on a 50-pair random sample**. Three-way agreement (Sonnet / Opus / GPT-5) gives a cheap signal on whether Sonnet's verdicts are biased toward Anthropic models under test.

If the three-way agreement is high (≥ 90% on the 50 sample), the self-preference concern is empirically reduced. If low, escalate the concern in the v2 design.

### Hard time budget

If by **Week 10** v1.3 is not done end-to-end (model runs, human coding, validation), stop scoping new features and ship what's there. The publication plan's risk is "v1.3 becomes the project" — this is the cap.

### Checkpoint gate

After Week 8, before any v2 work begins, an explicit **proceed / modify / rethink** decision based on v1.3 results:

- **Proceed:** Sonnet validates as defensible; rubric is stable; mining pipeline is solid → pre-register v2 mostly as designed.
- **Modify:** Sonnet barely passes or rubric needs more iteration → adjust v2 design, re-confirm with co-authors.
- **Rethink:** Sonnet fails or rubric reveals fundamental issues → step back, redesign before pre-reg.

## Infrastructure

### Gold-DB integration

v1.3 uses the existing gold-DB infrastructure:

- New `datasets` row for v1.3: `name='v1.3'`, mining window, source courts, frozen_at, git_commit.
- All 200 sampled rows go to `citation_rows` with `dataset_name='v1.3'`.
- Cases populated with `system` + `level` via `lookup_court()` at insert time.
- Verdicts get fresh `assessor_prompt_version='v1.3-sonnet-fulltext'` (and `-opus-fulltext` if the Opus fallback runs).
- Human verdicts get `source='human'`, `assessor_model='human'`, `assessor_prompt_version='v1.3-human-codebook-v1'`.

### v1 cohort backfill (one-shot)

Per the gold-DB plan's deferred items: v1's 130 cases have NULL `court_id` and NULL `year`. To enable tier-stratified comparison between v1 and v1.3 cohorts, run a one-shot fetch:

```python
for case in v1_cases_with_null_court_id:
    cluster = cl_client.get_cluster(case.cluster_id)
    case.court_id = cluster.court
    case.year = cluster.date_filed.year
    upsert_case(case)  # triggers lookup_court() to fill system + level
```

~130 CL lookups, ~30 minutes of work. Unlocks a published "v1 vs v1.3 by tier" figure for the writeup.

### Schema changes summary

1. `assessor_verdicts.source`: add `'human'` to the allowed values.
2. (Possibly): rename `gold.db` → `corpus.db` (the parked taxonomy question from commit `dfe178c`). **Deferred to a separate decision step** — not bundled with v1.3.

### Caches / efficiency

- **Build-side cache:** every cited case in v1.3 mining that's already in v1's `cases` table skips the API call. Mostly applicable for cases v1 already saw (SCOTUS, common circuit cases).
- **Score-side cache:** v1.3 verdicts won't collide with v1's cached Opus@20K because of the fresh `assessor_prompt_version`. Cross-cohort reuse is intentionally limited.
- **Drift sampling:** rolling 10 random pair re-checks per run, as in v1.

| Risk | Mitigation |
|---|---|
| State-court coverage is too poor for 25% allocation | Smoke-test contingency rule (Week 1); fall back to federal-only 33/33/33. |
