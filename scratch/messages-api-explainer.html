§1 · The primitive

One endpoint, several ways of using it

The Messages API is not a product with its own personality. It's a single HTTP endpoint that everything else is built on top of.

YOU SEND

A list of messages

System prompt, conversation so far, and — if you want — a document pasted right in. No memory of past calls, no state on Anthropic's side.

→
POST /v1/messages

The base primitive. Every pattern below — agent, CLI, batch job — is a particular way of calling this one endpoint, not a different technology.

→
YOU GET BACK

One completion

A single response. What happens next — another call, a tool result, nothing — is entirely up to whatever's calling the endpoint.

§2 · The mental model

Three switches, each set independently

Every way of running a batch of Claude requests is really just a position on three separate dials. Conflating them is the easiest mistake to make — and the one that quietly cost 5x here.

SWITCH 1

What the model can do mid-task

Agenticvs.Single-shot

Agentic means Claude gets tools — like Read a file — and decides turn by turn what to fetch, chaining many API calls with the conversation re-sent (and regrowing) each time. Single-shot means everything it needs is already in the prompt: one call, one answer.

SWITCH 2

How many requests run at once

Serialvs.Concurrent

Serial waits for each request to finish before starting the next — wall-clock time is the sum of everything. Concurrent fires several independent requests at once, so wall-clock shrinks toward the slowest single one.

SWITCH 3

When you need the answer

Real-timevs.Batch

Real-time answers in seconds. The Batches API queues up to 100K requests as one job, works through them within hours, and costs half as much — a trade of latency for a discount. It only works for self-contained, single-shot requests.

§4 · The wrinkle

Why the expensive corner can feel free

There's a fourth thing that gets tangled up with the three switches, and it's the reason the agentic corner is so easy to reach for.

Billing model ≠ architecture

claude -p and the Agent SDK sit in the exact same corner: agentic, and usually serial, and real-time. But claude -p is often run through a Pro or Max subscription — a flat monthly fee with usage included — while the Agent SDK, used to build something outside an interactive session, is typically billed per token through the API.

Run the same agentic, serial task through a subscription instead of a metered key, and it looks free. It isn't — the compute cost is the same either way. Subscription vs. metered is a question of who pays and how, completely separate from the three architecture switches above. It's exactly why the Agent SDK path here felt reasonable at first: it was subscription-covered, right up until it wasn't, and the $0.42-a-claim bill that had been there the whole time became visible.

§6 · The takeaway

Pick each switch on purpose

Three switches. Set each one deliberately.

01

Does the model actually need tools? If everything it needs fits in the prompt, agentic is pure overhead — you're paying for a negotiation the task never asked for.

02

Are the requests independent? If claim 12 doesn't depend on claim 11, there is no reason to run them serially. Concurrent costs the same per call and finishes in a fraction of the time.

03

Do you need the answer right now? If not, the Batches API is 50% off sitting unused for most workloads that already tolerate "later today."

04

Separate the bill from the architecture. A subscription can hide a metered cost; it can't make an inefficient architecture efficient. Check what you'd pay if it weren't covered.
