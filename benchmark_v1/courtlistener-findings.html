What we learned about CourtListener

Findings from running the Case Law Retrieval Benchmark v1, written for the Free Law Project / CourtListener team.

Companion to the main benchmark report · May 2026

What this is. While running a legal case retrieval benchmark, our pipeline made about 4,000 calls against CourtListener's APIs and mined 307 unique federal-district parenthetical citations, giving us a usable lens on what citation-lookup resolves and what it doesn't. The biggest pattern we noticed — a sharp F. Supp. 3d coverage cliff at 2020 — almost certainly lines up with what the free.law scanning project announced in April 2026 is set up to address. We're sharing the year-by-year breakdown in case it's useful for prioritization.

What this is not. A complaint. CL is the indispensable substrate for this kind of work; the reason we noticed these patterns is that we used CL heavily and most of it worked. We're happy to file GitHub issues with full reproducers, share data dumps, or help triage in any way that's useful.

Findings

data coverageF. Supp. 3d coverage stops at 2019 — sharp cliff, not gradual

From the 307 unique citations in our pool, we got a clean year-by-year view of how citation-lookup resolves district-court reporter cites. The pattern is striking and almost binary:

Every F. Supp. 3d cite from 2014–2019 resolves (14/14). Not one F. Supp. 3d cite from 2020 onward resolves (0/18). Same pattern across every citing district (DDC, ILND, MAD, etc.) — it's not a per-district issue, it's a reporter/year cutoff. Circuit reporters (F.2d / F.3d / F.4th) by contrast resolve at 93–100% from pre-1990 through 2024 in the same pool, dropping only at 2025 (the in-progress year). So the F. Supp. 3d cliff is roughly five years behind the circuit pipeline.

This is a citation-coverage issue, not an opinion-coverage one — the underlying opinions are largely in CL via PACER/RECAP; what's missing is the reporter-citation→opinion mapping. That's exactly what the scanning project is built to backfill, so we're sharing the year-by-year breakdown only in case the granularity is useful for prioritization. If a dump of the 18 unresolved 2020+ F. Supp. 3d cites would help validate scanning coverage, we have it ready.

The one figure here that isn't a scanning-project story: WL-only citations resolve at 0% via citation-lookup — 59 of them in our pool, ~19% of all citations, with no parallel reporter for citation-lookup to bind to. That's expected (Westlaw is paywalled), but the volume is worth noting for downstream tools: if a benchmark or verifier depends on CL, ~1 in 5 modern-district parentheticals will simply be unverifiable.

data coverageCitation-verifier fallback paths: where the misses actually live

For the 94 unique citations that citation-lookup couldn't resolve, we ran them through citation-verifier's full production fallback path (opinion-search, then RECAP for federal cites). Each result was hand-verified: did the cluster the verifier pointed to actually match the cited case, by reading the case name and confirming court + date?

Headline result: 75% of CL-misses (69 of 94) recover correctly when you go beyond citation-lookup. The status flag the verifier returns is meaningfully predictive of trustworthiness: opinion-search + LIKELY_REAL is 100% accurate (29 of 29) — a trust-it tier — while everything else (POSSIBLE_MATCH from either path, or anything from RECAP) hovers around 69–70% accuracy and warrants manual confirmation.

WL recovery is much better than citation-lookup-alone suggests. WL-only citations resolve at 0% via citation-lookup (no parallel reporter to bind to), but 47 of 59 (80%) recover correctly through the fallback path — opinion-search and RECAP can find the underlying case via name + year + court when citation-lookup gives up. So a downstream tool that uses only citation-lookup is dropping ~1 in 5 modern-district parentheticals as unverifiable when ~4 in 5 of those are actually findable.

Cases that may genuinely not be in CL at all. We hand-verified 7 pre-cliff reporter misses (the 7 distinct citation-lookup NOT_FOUND items in the well-covered reporter buckets — F.3d, F. App'x, F.4th pre-2025, F. Supp. 1st-series). Of those 7, only one is confirmed genuinely missing: Brown v. Osmundson, 38 F.4th 545 (7th Cir. 2022) — published on the 7th Cir.'s website but with no cluster in CL by name, citation, or docket. Worth a closer look at the 7th Cir. ingestion pipeline. The other ~70 misses (F. Supp. 3d 2020+, recent F. App'x, recent F.4th, WL-only, LEXIS) are likely a mix of the F. Supp. 3d cliff pattern (in CL by name, no reporter mapping) and the WL/LEXIS paywall, but we did not hand-verify each.

The full audit data (annotations, paths, matched cluster URLs) is at benchmark_v1/_all_cl_misses.csv.

minorNortheastern reporter (N.E.2d/N.E.3d) coverage drops to 79%, smaller for Mass.-specific reporters

Of 14 unique Northeastern-reporter citations in our pool (mostly Massachusetts state cases), 11 resolved via citation-lookup (79%) — meaningfully lower than Atlantic / Pacific / Southwestern (all 100%) and Southern Reporter cases. For Mass.-specific state reporters (e.g. 496 Mass. 197) the rate is 1/2 (n too small to interpret on its own). Both buckets are small, so this is a hint, not a robust pattern — but if Northeastern indexing has a known gap relative to other regional reporters, our data is consistent with it.

minorcitation-lookup clusters returned with case_name empty

When citation-lookup resolves a citation, the returned cluster object frequently has "case_name": null even when the cluster's absolute_url contains the slug (e.g. /opinion/145814/virginia-v-moore/) and the cluster ID resolves correctly via the /clusters/<id>/ endpoint with full case-name population.

For callers doing name-matching (which we are — to detect cases like "this citation is real but the model named the wrong case"), this means a follow-up /clusters/<id>/ fetch to get the actual name. Not a bug per se — just an observation that two API endpoints return slightly different shapes for the same cluster, and the citation-lookup shape is more minimal than callers might expect. Either populating case_name in the citation-lookup response, or documenting that callers need to follow up via /clusters/<id>/, would help.

minoropinion text retrieval needs a 6-field fallthrough

To pull opinion text from a cluster, we have to try plain_text first, then fall through to html, html_columbia, html_with_citations, html_lawbox, and xml_harvard, stripping markup along the way. Some opinions populate one field; some populate another; some populate several inconsistently.

This is presumably a consequence of CL's import history (different sources populate different fields). Not asking for normalization — that'd be a huge project — but a one-paragraph "how to robustly get the text of an opinion" in the API docs, with the recommended fallthrough order, would save every downstream tool from inventing its own.

minoreyecite extracts each citation in a parenthetical multiple times

While mining parentheticals from federal-district opinions, we found that eyecite returns the same citation in multiple resolved forms within a single source opinion (full reference + supra/id-style short cites). Our pre-dedup mining produced ~10× the entries we expected, all on the same (source opinion, citation) pair. Anyone doing eyecite-based citation extraction should expect to dedup on (source_opinion_id, citation_text, surrounding_paren) before downstream processing — otherwise downstream pools / sample frames will heavily over-weight cases that happen to be cited multiple times in the same opinion. We don't think this is wrong behavior on eyecite's part (it's a feature for some callers), just worth a note in the docs for consumers building corpora.

A note on what we left out: while running this benchmark we also hit citation-lookup's response-truncation behavior on large batches (we sent 600 citations in one call and got entries for only ~250). We understand this is already known to FLP; we worked around it by chunking to 150 per request. Mentioning only so the audit trail is complete.

What we'd find useful (wishlist)

Not asks — just things that would simplify benchmarks like ours:

  • An "is this real?" endpoint that fans out across multiple resolution paths AND uses all available metadata. Today, citation-lookup is fast but conservative; opinion-search is more permissive but slower. We hand-audited the 94 unique CL-misses in our pool by passing them through citation-verifier's full opinion-search fallback. Two empirical findings shaped the ask: (1) the fallback recovered ~70% of the misses overall — these mostly weren't coverage gaps but cases where citation-lookup couldn't resolve them given just (volume, reporter, page); (2) when we re-ran the same audit passing court info (e.g., (7th Cir.)) to opinion-search, 6 of 26 URL-changed cases got newly-correct disambiguations and zero regressed — same-name-same-reporter-slot ambiguities like Azumi LLC (cited from D. Mass., but a different Azumi LLC case exists in S.D. Fla. at the same reporter slot) only resolve correctly when court is part of the search. A unified existence-check endpoint that tries cite-lookup first, falls through to a metadata-aware opinion-search, would let downstream tools have one canonical "real or not" oracle.
  • A bulk-resolution endpoint that handles 1,000+ citations cleanly. Would let benchmark scoring be a single API call instead of N batched calls.
  • Per-citation court-tier metadata in citation-lookup's response. Would let benchmark consumers stratify by tier without separately fetching /clusters/<id>/ for each.
  • A test-data endpoint or fixture set for citation-lookup. When integrating CL into a tool, a small set of known-good citations (covering each reporter, each tier, edge cases) would be invaluable for regression tests.

What we have on hand

If any of these are useful, we have:

  • The full audited miss CSV (benchmark_v1/_all_cl_misses.csv): 94 unique citation-lookup misses with hand-verified ground truth, fallback path, and matched cluster URL — the data behind Table 3 above.
  • The 18 unresolved 2020+ F. Supp. 3d citations as a separate dump on request — useful for validating scanning-project coverage.
  • The mining pool (benchmark_v1/_raw_pool.json) with all 307 unique citations, verification statuses, and per-citation metadata.
  • The open-source citation-verifier codebase, which uses CL heavily.

The citation-verifier project itself exists because of CourtListener — happy to help with issue triage, test cases, sample data, or just talking through findings.
