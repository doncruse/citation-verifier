What we learned about CourtListener

Findings from running the Case Law Retrieval Benchmark v1, written for the Free Law Project / CourtListener team.

Companion to the main benchmark report · May 2026

What this is. While running a 600-cell legal case retrieval benchmark, our pipeline made about 4,000 calls against CourtListener's APIs. We mined and verified 3,070 federal-district parentheticals along the way, which gave us a usable lens on what citation-lookup resolves and what it doesn't. The biggest pattern we noticed — a sharp F. Supp. 3d coverage cliff at 2020 — almost certainly lines up with what the free.law scanning project announced in April 2026 is set up to address. We're sharing the year-by-year breakdown in case it's useful for prioritization.

What this is not. A complaint. CL is the indispensable substrate for this kind of work; the reason we noticed these patterns is that we used CL heavily and most of it worked. We're happy to file GitHub issues with full reproducers, share data dumps, or help triage in any way that's useful.

Findings

data coverageF. Supp. 3d coverage stops at 2019 — sharp cliff, not gradual

Of the 3,070 parentheticals we mined from federal-district opinions filed in early 2026, we got a clean year-by-year view of how citation-lookup resolves district-court reporter cites. The pattern is striking and almost binary:

Every F. Supp. 3d cite from 2014–2019 in our pool resolves. Not one F. Supp. 3d cite from 2020 onward resolves. Same pattern across every citing district (DDC, ILND, MAD, etc.) — it's not a per-district issue, it's a reporter/year cutoff. Circuit reporters (F.2d / F.3d / F.4th) by contrast resolve at 93–100% from pre-1990 through 2024 in the same pool, dropping only at 2025 (the in-progress year). So the F. Supp. 3d cliff is roughly five years behind the circuit pipeline.

This is a citation-coverage issue, not an opinion-coverage one — the underlying opinions are largely in CL via PACER/RECAP; what's missing is the reporter-citation→opinion mapping. That's exactly what the scanning project is built to backfill, so we're sharing the year-by-year breakdown only in case the granularity is useful for prioritization. If a dump of the 180 unresolved 2020+ F. Supp. 3d cites would help validate scanning coverage, we have it ready.

The one figure here that isn't a scanning-project story: WL-only citations resolve at 0% — 590 of them in the pool, ~19% of all parentheticals in modern district opinions, with no parallel reporter to fall back on. That's expected (Westlaw is paywalled), but the volume is worth noting for downstream tools: if a benchmark or verifier depends on CL, ~1 in 5 modern-district parentheticals will simply be unverifiable.

For completeness, we also re-checked each NOT_FOUND citation against opinion-search (the next fallback path) and saw a clean split: F.3d misses recovered at 100% (20/20), but F.4th 2025+ misses, F. Supp. 3d 2020+ misses, and WL misses recovered at 0% (260/260 stayed NOT_FOUND). Citation-lookup is doing some work that opinion-search isn't redundant with, even when both endpoints have access to the same underlying data.

minorcitation-lookup clusters returned with case_name empty

When citation-lookup resolves a citation, the returned cluster object frequently has "case_name": null even when the cluster's absolute_url contains the slug (e.g. /opinion/145814/virginia-v-moore/) and the cluster ID resolves correctly via the /clusters/<id>/ endpoint with full case-name population.

For callers doing name-matching (which we are — to detect cases like "this citation is real but the model named the wrong case"), this means a follow-up /clusters/<id>/ fetch to get the actual name. Not a bug per se — just an observation that two API endpoints return slightly different shapes for the same cluster, and the citation-lookup shape is more minimal than callers might expect. Either populating case_name in the citation-lookup response, or documenting that callers need to follow up via /clusters/<id>/, would help.

minoropinion text retrieval needs a 6-field fallthrough

To pull opinion text from a cluster, we have to try plain_text first, then fall through to html, html_columbia, html_with_citations, html_lawbox, and xml_harvard, stripping markup along the way. Some opinions populate one field; some populate another; some populate several inconsistently.

This is presumably a consequence of CL's import history (different sources populate different fields). Not asking for normalization — that'd be a huge project — but a one-paragraph "how to robustly get the text of an opinion" in the API docs, with the recommended fallthrough order, would save every downstream tool from inventing its own.

A note on what we left out: while running this benchmark we also hit citation-lookup's response-truncation behavior on large batches (we sent 600 citations in one call and got entries for only ~250). We understand this is already known to FLP; we worked around it by chunking to 150 per request. Mentioning only so the audit trail is complete.

What we'd find useful (wishlist)

Not asks — just things that would simplify benchmarks like ours:

  • An "is this real?" endpoint that fans out across multiple resolution paths. Today, citation-lookup is fast but conservative; opinion-search is more permissive but slower. Empirically: of the 20 F.3d citations that citation-lookup missed in our pool, opinion-search rescued 20/20 (100%) — those weren't coverage gaps, just parser misses. A unified existence-check endpoint that tries cite-lookup first, then falls through to opinion-search, would let downstream tools have one canonical "real or not" oracle without each writing its own fallback chain.
  • A bulk-resolution endpoint that handles 1,000+ citations cleanly. Would let benchmark scoring be a single API call instead of N batched calls.
  • Per-citation court-tier metadata in citation-lookup's response. Would let benchmark consumers stratify by tier without separately fetching /clusters/<id>/ for each.
  • A test-data endpoint or fixture set for citation-lookup. When integrating CL into a tool, a small set of known-good citations (covering each reporter, each tier, edge cases) would be invaluable for regression tests.

What we have on hand

If any of these are useful, we have the full 3,070-row mined pool with verification status, year, and tier classification (benchmark_v1/_raw_pool.json); the 180 unresolved post-2019 F. Supp. 3d citations as a separate dump on request; and the open-source citation-verifier codebase, which uses CL heavily.

The citation-verifier project itself exists because of CourtListener — happy to help with issue triage, test cases, sample data, or just talking through findings.
